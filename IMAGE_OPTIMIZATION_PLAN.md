# 시공사례 이미지 최적화 계획

`IMAGE_MAPPING.md` 6번(이미지 최적화 후보)에서 표시만 하고 실행하지 않은 작업의 구체 실행
계획이다. 대상은 `public/images/works/` 139장(약 411MB)이며, 목표는 "웹사이트 배포 적합
수준"(저장소/배포 용량과 페이지 로딩 속도에 무리가 없는 수준)으로 낮추는 것이다.

## 0. 현황 (실측)

| 그룹 | 폴더 | 장수 | 용량 | 형식 | 원본 특성 |
|---|---|---|---|---|---|
| A | `ihwa-livestock-fattening-renovation` | 57 | 233MB | jpg/png | 휴대폰 원본 4000×2250급, 장당 4~5MB |
| B | `andong-piglet-house-construction`, `andong-fattening-house-reconstruction`, `andong-sow-house-reconstruction`, `hwaseong-farm-reconstruction` | 28 | 140MB | png 다수 | 사진을 PNG로 저장(무손실) — 그래픽이 아니라 카메라 원본 |
| C | 그 외 8개 케이스 (`nonghyup-*`, `dondonfarm-*`, `jindo-hyodon`, `hoengseong-fire-restoration`, `ihwa-livestock`) | 54 | 약 37.5MB | jpg | 평균 0.5~2.8MB/장, 비교적 정상 범위 |
| D | `logo/` | 5 | 1.6MB | png/jpg | 인쇄용 해상도(6332~6486px) |
| E | `hero/` | 1 | 1.4MB | png | 1405×567 — 해상도 자체가 낮음(별도 이슈) |

전체 jpg 106장 / png 33장. PNG 33장은 대부분 그룹 B(사진을 PNG로 저장한 경우)이며, 로고 3장만
투명배경이 필요한 진짜 그래픽 PNG다.

**런타임 환경 확인**: `next.config.ts`에 커스텀 `images.loader` 설정이 없고, `sharp`가
`node_modules`에 설치되어 있다(Next.js 기본 Image Optimization API가 동작 가능한 상태).
즉 `next/image`가 요청 시점에 `sizes`/디바이스 폭에 맞춰 자동으로 리사이즈·포맷 변환(WebP/AVIF)
을 수행한다 — 따라서 이번 계획은 **카드용/갤러리용 파일을 별도로 미리 구워내는 대신, 위치별로
필요한 "최대 표시 크기"에 맞춘 마스터 1장만 만들고 나머지는 Next에 맡기는 방식**으로 잡는다.

## 1. 리사이즈 규칙

각 work 폴더의 이미지가 실제로 쓰이는 곳(`WorkCard.tsx`, `WorkGallery.tsx`)의 `sizes` 속성을
기준으로, 그 중 **가장 큰 표시 폭**을 커버하는 단일 마스터 치수를 정한다.

| 용도 | 코드 위치 | 실사용 `sizes` | 마스터 긴 변 기준 |
|---|---|---|---|
| 시공사례 사진 (works/*) | `WorkGallery.tsx` 메인 뷰어/라이트박스 | `800px` / `100vw` | **1600px** (업스케일 금지, 원본이 더 작으면 원본 유지) |
| 로고 (logo/*) | `Header`/`Footer`, 실제 표시 폭 150~240px | — | **720px** (표시 폭의 약 3배, 2~3x DPR 대응) |
| 메인 비주얼 (hero/*) | Hero 배경 풀블리드 | `100vw` | 현재 1405×567로 풀블리드 배경 기준에도 못 미침 → **리사이즈 보류**, 고객에게 고해상도 원본 재요청을 우선 진행 (`IMAGE_MAPPING.md` 5-4 참고) |

공통 규칙:
- 원본보다 큰 치수로 키우지 않는다(업스케일 금지).
- 종횡비는 그대로 유지한다(크롭은 화면에서 `object-cover`로 처리하므로 리사이즈 단계에서는 하지 않음).
- 리사이즈 시 EXIF Orientation을 픽셀에 반영한 뒤 메타데이터(EXIF/ICC 등)는 제거한다.
- 카드(`WorkCard.tsx`, `33vw`/`50vw`/`85vw`)와 썸네일 스트립(`96px`)은 별도 파일을 만들지
  않고, 1600px 마스터를 Next.js Image Optimization API가 요청 폭에 맞춰 자동으로 줄인
  버전을 서빙·캐시하도록 한다(§3 참고).
- `next.config.ts`의 `images.deviceSizes`/`images.imageSizes`를 실제 코드에 등장하는 폭
  (`96, 400, 800, 1600` 등)에 맞게 좁혀서, Next가 불필요하게 다양한 중간 크기를 캐시하지
  않도록 조정한다.

## 2. WebP 변환 규칙

- **사진성 콘텐츠는 원본 형식(jpg/png) 무관하게 WebP로 통일**한다. 그룹 B의 PNG 28장은
  그래픽이 아니라 카메라 원본을 무손실로 저장한 것뿐이라 무손실을 유지할 이유가 없다 —
  리사이즈와 함께 손실 압축 WebP로 전환한다.
- 품질은 **q80 기준**으로 시작하고(허용 범위 q75~85), 변환 후 샘플 이미지를 시각 검수해
  깨짐이 보이면 그룹 단위로 +5 조정한다.
- 변환 시 메타데이터(EXIF/ICC/GPS 등)는 모두 제거한다.
- **투명배경이 필요한 로고 3종**(`logo-mark.png`, `logo-full.png`, `logo-full-alt.png`)은
  마스터를 PNG로 유지하되 치수만 720px로 축소한다. 흰배경 로고 2종(jpg)은 사진과 동일하게
  WebP 변환 대상이다.
- 마스터 파일 자체를 WebP로 바꾸는 것과 별개로, 브라우저에 실제로 내려가는 포맷은 Next.js
  Image Optimization API가 `Accept` 헤더에 따라 AVIF/WebP/원본 중 자동 협상한다. 그럼에도
  마스터를 WebP로 미리 바꾸는 이유는 **저장소·배포 용량**과 **최초 요청 시 변환 비용**을
  줄이기 위함이다(원본이 5MB짜리 PNG면 첫 요청마다 그만큼을 읽어 변환해야 한다).

## 3. 썸네일 규칙

- 각 work 폴더의 `01.*`을 대표 썸네일(카드/하이라이트용 대표컷)로 유지한다 — `IMAGE_MAPPING.md`
  에서 이미 잠정 합의된 가정이며, 실제 대표컷 재선정은 콘텐츠 작업 시 별도 검토.
- 별도의 `*-thumb.webp`, `*-strip.webp` 같은 파생 파일은 만들지 않는다. §1에서 만든 1600px
  WebP 마스터 하나를 `WorkCard.tsx`(`fill` + `sizes="33vw/50vw/85vw"`)와 `WorkGallery.tsx`
  썸네일 스트립(`sizes="96px"`)이 공통으로 참조하면, Next.js Image Optimization API가 각
  요청 폭에 맞는 축소판을 자동 생성·캐시한다. 파일 중복 관리와 `lib/mock-works.ts`의 데이터
  모델 변경(이미지당 URL 2~3개 관리) 부담을 피하기 위한 결정이다.
- 단, 위 방식은 `next start`(자체 서버, sharp 동작) 또는 Vercel처럼 Image Optimization API가
  살아있는 배포 환경을 전제로 한다. 순수 정적 export(`next export`)로 배포할 계획이라면 이
  전제가 깨지므로 사전에 배포 방식을 확인해야 한다.

## 4. 원본 보관 전략

- `customer-assets/`(고객이 제공한 최초 원본, 152장 + 영상 3개)은 손대지 않고 그대로 둔다.
  단, 이미 git 커밋(`54e8afc`)에 포함되어 저장소 자체가 무겁다는 점은 별개 이슈로 남는다 —
  git 히스토리까지 가볍게 하려면 BFG/`git filter-repo` 같은 히스토리 재작성이 필요한데, 이는
  파괴적 작업이라 이번 계획 범위에서 제외하고 별도로 사용자 승인을 받아 진행할 사안으로 둔다.
- `public/images/works/`의 현재 139장(정규화는 됐지만 미압축인 "준원본")은 §1~2 처리의
  **입력**으로만 쓰고, 처리 후에는 `public/` 밖의 비-서빙 경로(예: `assets/originals/works/`)
  로 옮긴다. `public/`에는 최적화된 WebP 마스터만 남겨 배포 결과물에 포함되는 용량을 최소화한다.
- 보관 우선순위: ① `customer-assets`(최초 원본, 무손실) → ② `assets/originals`(정규화된
  준원본, 무손실) → ③ `public/images`(배포용 WebP 마스터). ①·②는 git 저장소보다 외부
  스토리지(클라우드 드라이브/NAS 등)에 이중 보관하는 것을 권장하며, 이번 단계에서는 위치
  이동 계획만 세우고 실제 외부 백업 수단은 사용자 확인 후 결정한다.
- 향후 크롭/품질 재조정이 필요해지면 ②를 다시 입력으로 써서 재처리하면 되므로, ③만 git에
  있어도 복구 가능성은 유지된다.

## 5. 예상 용량 감소율

| 그룹 | 현재 용량 | 처리 | 예상 감소율 | 예상 결과 용량 |
|---|---|---|---|---|
| A. 이화축산 비육사재축 (57장) | 233MB | 1600px 리사이즈 + WebP q80 | 약 90% | ~23MB |
| B. 안동 3종 + 화성 (28장, PNG) | 140MB | 1600px 리사이즈 + WebP q80 (PNG→WebP 전환 효과 큼) | 약 95% | ~7MB |
| C. 나머지 8개 케이스 (54장) | 37.5MB | 1600px 리사이즈(대부분 원본이 이미 1600px 근접/이하) + WebP q80 | 약 65~70% | ~12MB |
| D. 로고 (5장) | 1.6MB | 720px 리사이즈 + (투명 3장 PNG 유지 / 2장 WebP) | 약 65% | ~0.6MB |
| E. 메인 비주얼 (1장) | 1.4MB | 리사이즈 보류, 포맷 변환만 | 약 10% | ~1.3MB |
| **합계** | **약 411MB** | | **약 88~90%** | **약 40~45MB** |

비고:
- C그룹은 원본이 이미 어느 정도 합리적인 크기라 A·B만큼의 극단적 절감은 아니지만, 그래도
  60% 이상 절감이 기대된다.
- 위 수치는 동일 계열 사진(휴대폰 카메라, 유사 해상도)의 일반적인 JPEG/PNG → 리사이즈+WebP
  절감률을 근거로 한 추정치다. 실제 실행 단계에서 그룹별로 5~10장씩 파일럿 변환 후 실측치로
  보정하는 것을 권장한다.
- 목표였던 "411MB → 웹 배포 적합 수준"은 약 40~45MB 선으로 달성 가능할 것으로 예상되며,
  이는 일반적인 정적 사이트 이미지 자산 규모로 git 클론/배포 속도에 부담이 거의 없는 수준이다.

## 6. 실행 메모 (이번 계획에는 미포함, 다음 단계 참고용)

- `sharp`는 이미 `next`의 의존성으로 설치되어 있으나 직접 호출하려면 `package.json`에
  devDependency로 명시 추가가 필요하다.
- 1회성 변환 스크립트(`scripts/optimize-images.mjs` 등)를 작성해 그룹별로 일괄 실행하고,
  변환 전/후 샘플을 시각 비교한 뒤 `public/images/works`를 교체하는 순서를 권장한다.
- 이번 문서는 계획 수립까지만 다루며, 실제 스크립트 작성·실행은 사용자 승인 후 별도 작업으로
  진행한다.
