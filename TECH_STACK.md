# 기술 스택 선정안 (Tech Stack Decision)

> PRD.md 기준 평가. 평가 기준: ① SEO 대응력 ② 관리자/이미지업로드/메일발송 구현 난이도 ③ 유지보수 용이성(향후 인력 교체·확장 대응) ④ 비용 효율성(약 10페이지 규모 기업 홍보 사이트에 과한 스펙인지)

---

## 1. 채택 여부 요약

| 후보 | 채택 여부 | 역할 |
|---|---|---|
| 1. Next.js | ✅ 채택 | 프론트엔드 프레임워크 (SSR/SSG, API Route, SEO) |
| 2. Tailwind CSS | ✅ 채택 | 스타일링 (반응형/모바일 최적화) |
| 3. Supabase | ✅ 채택 (Pro 권장) | DB(문의·시공사례 데이터) + Auth(관리자 로그인) + Storage(이미지 업로드) |
| 4. Resend | ✅ 채택 | 문의 접수 시 회사 메일 자동 발송(트랜잭션 메일) |
| 5. Vercel | ✅ 채택 (Pro 필요) | 배포/호스팅 + CDN |

**결론: 5개 후보 모두 채택.** 단, Supabase·Vercel은 무료 플랜이 운영 안정성·약관상 부적합하여 유료 플랜 사용을 전제로 비용을 산정함(아래 3장 참고).

## 2. 후보별 선정 이유

### 2-1. Next.js — ✅ 채택
**선정 이유**
- SEO가 핵심 요구사항인데, Next.js는 SSG(정적 생성)/ISR로 메인·회사소개·사업분야 등 정적성 강한 페이지를 빌드 타임에 HTML로 미리 만들어 크롤러가 완전한 콘텐츠를 즉시 읽을 수 있게 함 (PRD 5장 SEO 요구사항: Core Web Vitals, 구조화 데이터, sitemap 자동화와 직결)
- 시공사례 상세(`/works/:id`)처럼 관리자가 콘텐츠를 추가하는 동적 페이지는 ISR(Incremental Static Regeneration)로 처리하면 신규 등록 시 자동 재생성 — 별도 캐시 무효화 로직 불필요
- `next/image`로 이미지 자동 최적화(WebP 변환, lazy loading) — PRD가 요구하는 이미지 최적화·LCP 기준을 프레임워크 차원에서 해결
- API Route(또는 Route Handler)로 문의폼 제출·메일발송·관리자 인증 미들웨어를 별도 백엔드 서버 없이 한 프로젝트 안에서 처리 → 인프라 단순화(유지보수 용이성 직결)
- React 생태계 기반이라 향후 개발자 채용·인수인계가 쉬움(국내 채용 시장에서 가장 보편적인 프레임워크)

**대안 검토**: 순수 React+Vite(SPA)는 SEO에 불리(크롤러가 빈 HTML을 받음), WordPress는 빠르지만 PRD의 "관리자 이미지 교체, 커스텀 문의 프로세스"를 코드 레벨로 세밀히 제어하기엔 플러그인 의존도가 높아 장기 유지보수 비용이 오히려 커짐 → Next.js가 이 규모(10페이지, 커스텀 관리자 기능)에 가장 적합

### 2-2. Tailwind CSS — ✅ 채택
**선정 이유**
- PRD의 핵심 비기능 요구사항인 "반응형/모바일 최적화"를 breakpoint 유틸리티(`sm:`, `md:`, `lg:`)로 빠르고 일관되게 구현
- 별도 CSS 파일/네이밍 규칙 설계가 필요 없어 약 10페이지 규모에서 디자인 시스템 구축 비용 대비 효율이 높음
- shadcn/ui 같은 Tailwind 기반 컴포넌트 라이브러리와 결합하면 관리자 페이지의 폼/테이블/모달을 빠르게 구현 가능 → 개발 기간 단축
- 클래스가 컴포넌트 파일에 함께 위치해 향후 유지보수 담당자가 바뀌어도 "이 스타일이 어디서 오는지" 추적이 쉬움(전역 CSS 충돌 이슈가 적음)

**유의점**: 유틸리티 클래스가 많아지면 JSX 가독성이 떨어질 수 있으므로 반복되는 패턴은 컴포넌트로 추출 권장(개발 가이드라인으로 관리)

### 2-3. Supabase — ✅ 채택 (Pro 플랜 권장)
**선정 이유**
- PRD 관리자 기능(로그인, 문의내역 관리, 시공사례 CRUD, 이미지 관리)에 필요한 3요소(DB+Auth+Storage)를 하나의 서비스로 제공 → 별도 인증서버·파일서버 구축/운영 비용 제거
- Postgres 기반이라 문의 데이터(Contacts), 시공사례(Works), 카테고리 등 관계형 데이터 모델링에 적합하고, 향후 기능 확장(필터/통계) 시 SQL로 유연하게 대응 가능
- Storage는 이미지 업로드 즉시 CDN URL을 반환하므로 관리자 이미지 교체 기능(메인비주얼/시공사례 갤러리)을 그대로 구현 가능
- Auth는 이메일/비밀번호 기반 관리자 로그인을 자체 구현(JWT 발급, bcrypt 해싱, 세션 만료)보다 안전하고 빠르게 적용 가능

**유의점(중요)**: Supabase **Free 플랜은 7일간 활동이 없으면 프로젝트가 자동 일시정지(pause)** 됨. 상시 운영되는 기업 홈페이지에서 이 상태가 되면 관리자 로그인 실패·문의 저장 실패로 이어질 수 있어 **운영 단계에서는 Pro 플랜 필수**로 판단함(자동 일시정지 해제 + 일일 백업 제공).

### 2-4. Resend — ✅ 채택
**선정 이유**
- PRD 4장 "문의 시 회사 메일 자동 발송"을 구현하는 가장 단순한 방법. Nodemailer+개인 Gmail SMTP 조합보다 스팸 처리율이 낮고, Next.js 공식 가이드와의 통합이 쉬움
- 발신 도메인(예: `noreply@회사도메인.com`) 인증(DKIM/SPF)을 지원해 메일이 스팸함으로 분류될 가능성을 낮춤 — 문의 메일이 누락되면 곧 영업 손실로 이어지는 만큼 신뢰성이 중요
- 무료 플랜(월 3,000건, 1일 100건)이 기업 홍보 사이트의 문의량(통상 월 수십~수백 건) 대비 충분히 여유 있음

**유의점**: 별도 도메인 DNS에 SPF/DKIM 레코드 추가 필요(1회성 설정, 추가 비용 없음)

### 2-5. Vercel — ✅ 채택 (Pro 플랜 필요)
**선정 이유**
- Next.js 제작사이므로 ISR, Image Optimization, Edge 캐싱 등 Next.js 고유 기능이 별도 설정 없이 완전히 동작 — 다른 호스팅(AWS, Netlify)으로 가면 일부 기능을 직접 구성해야 해 유지보수 부담이 커짐
- Git 연동 자동 배포(PR 단위 프리뷰 배포)로 향후 유지보수 담당자가 바뀌어도 배포 프로세스가 표준화되어 있음
- 글로벌 CDN으로 정적 페이지 응답속도 확보 → Core Web Vitals(LCP) 개선에 직접 기여

**유의점(중요)**: Vercel **Hobby(무료) 플랜은 약관상 비상업적 용도로 한정**되어 있어, 기업 홍보 목적의 상업적 사이트는 **Pro 플랜 사용이 약관 준수 측면에서 필요**함.

## 4. 전체 아키텍처 개요

```
[사용자 브라우저]
    │  HTTPS
    ▼
[Vercel — Next.js 앱]
    ├─ SSG/ISR 페이지 (메인, 회사소개, 사업분야, 시공사례 목록/상세) → SEO 대응
    ├─ Route Handler /api/contact → Resend 호출(회사 메일 발송) + Supabase DB 저장
    ├─ Route Handler /api/admin/* → Supabase Auth 인증 검증 후 CRUD 처리
    └─ next/image → Supabase Storage 이미지 최적화 렌더링
              │
              ▼
[Supabase]
    ├─ Postgres DB (Works, Categories, Contacts 테이블)
    ├─ Auth (관리자 계정, JWT 세션)
    └─ Storage (메인비주얼/시공사례/사업자등록증 이미지, CDN 제공)
              │
              ▼
[Resend] → 문의 접수 시 회사 담당 메일함으로 발송
```

## 5. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| Supabase Free 플랜 자동 일시정지 | 관리자 로그인/문의 저장 실패 | 운영 전환 시 Pro 플랜 필수 적용 |
| Vercel Hobby 상업적 이용 제한 | 약관 위반 소지 | 운영 전환 시 Pro 플랜 필수 적용 |
| Resend 무료 한도(월 3,000건/일 100건) 초과 | 메일 발송 실패 | 문의량 모니터링 후 유료 플랜($20/월~) 전환 검토 |
| 이미지 업로드 용량 급증 | Storage 비용 증가 | 업로드 시 자동 리사이즈/WebP 변환 적용(PRD 3-3 요구사항)으로 평균 용량 최소화 |
